# Feedback Process

**피드백 수집 및 처리 프로세스** | User Feedback Collection & Processing

이 문서는 KooDTX 앱의 사용자 피드백을 수집하고 제품 개선에 반영하는 전체 프로세스를 설명합니다.

---

## 📬 피드백 수집 채널

### 1. Google Play 리뷰

**장점:**
- 공개적으로 가시적
- 평점이 앱 다운로드에 영향
- Google Play가 자동 수집

**수집 방법:**
- Play Console > 평점 및 리뷰
- 이메일 알림 활성화 (새 리뷰 즉시 알림)

**주의:**
- 24-48시간 내 답변 (특히 1-2점 리뷰)
- 공개 답변이므로 전문적으로 작성

---

### 2. 이메일 (support@koodtx.com)

**장점:**
- 상세한 문의 가능
- 개인적 소통
- 첨부 파일 (스크린샷, 로그) 가능

**수집 방법:**
- Gmail 계정으로 수신
- 라벨로 분류 (`bug`, `feature-request`, `question`)

**응답 시간:**
- 24-48시간 내 초기 응답 (평일 기준)

---

### 3. 앱 내 피드백 (Phase 249)

**장점:**
- 사용자가 쉽게 접근
- 자동으로 디바이스 정보 포함
- 즉시 전송

**구현:**
- `FeedbackScreen.tsx` 참고
- 설정 > 피드백 보내기

**데이터:**
- 피드백 타입 (버그, 기능 요청, 일반)
- 메시지
- 이메일 (선택)
- 디바이스 정보 (자동)

---

### 4. Analytics & Crash Reports

**장점:**
- 사용자가 직접 말하지 않은 문제 발견
- 정량적 데이터

**수집 방법:**
- **Sentry**: 크래시 로그, 오류 추적
- **Firebase Analytics**: 사용 패턴, 퍼널 분석
- **Play Console Android vitals**: 크래시율, ANR

**분석:**
- 주간 리포트 작성 (docs/DAILY_MONITORING_CHECKLIST.md)

## 🔄 피드백 처리 워크플로우

### 1단계: 수집 및 분류 (Daily)

**작업:**
1. 모든 채널에서 새로운 피드백 확인
   - Play 리뷰
   - 이메일
   - 앱 내 피드백

2. 각 피드백을 다음으로 분류:
   - **버그 리포트**: 기능이 제대로 작동하지 않음
   - **기능 요청**: 새로운 기능 또는 개선 제안
   - **일반 문의**: 사용법, 질문 등
   - **칭찬**: 긍정적 피드백

3. GitHub Issue 생성 (필요 시)
   - 버그 → `.github/ISSUE_TEMPLATE/bug_report.md`
   - 기능 요청 → `.github/ISSUE_TEMPLATE/feature_request.md`

---

### 2단계: 초기 응답 (24-48시간 내)

**템플릿 사용:**
- `docs/SUPPORT_EMAIL_TEMPLATES.md` 참고

**응답 원칙:**
- **공감**: 사용자의 불편/제안에 공감 표현
- **명확한 설명**: 문제 원인 또는 제안 검토 계획 설명
- **후속 조치**: 다음 단계 안내 (수정 예정, 추가 정보 요청 등)

**예시:**
```
안녕하세요 [이름]님,

KooDTX 팀입니다.
[문제/제안]에 대한 피드백 주셔서 감사합니다.

말씀하신 [내용]은 [원인/계획]입니다.
[해결 방법 또는 검토 계획]

추가 문의사항이 있으시면 언제든지 연락 주세요.

감사합니다.
KooDTX 지원팀
```

---

### 3단계: 분석 및 우선순위화 (Weekly)

**주간 피드백 리뷰:**
1. 지난 주 수집된 모든 피드백 검토
2. 공통 패턴 찾기
   - 같은 버그를 여러 사용자가 리포트?
   - 같은 기능을 여러 사용자가 요청?

3. 우선순위 결정
   - `docs/FEATURE_PRIORITIZATION.md` 프레임워크 사용
   - Impact vs Effort Matrix로 평가

**리뷰 템플릿:**
```markdown
# Weekly Feedback Review - Week of [YYYY-MM-DD]

## 📊 통계
- 총 피드백: XX건
- 버그 리포트: XX건
- 기능 요청: XX건
- 일반 문의: XX건
- 칭찬: XX건

## 🔥 Hot Topics
1. [주제 1]: XX건 (예: GPS 정확도 낮음)
2. [주제 2]: XX건
3. [주제 3]: XX건

## 🐛 주요 버그
1. [버그 1]: [설명] - Priority: [Critical/High/Medium/Low]
2. [버그 2]: [설명] - Priority: [...]

## 💡 주요 기능 요청
1. [기능 1]: [설명] - Priority: [Critical/High/Medium/Low]
2. [기능 2]: [설명] - Priority: [...]

## ✅ Action Items
- [ ] Critical 버그 즉시 수정 (Hotfix 고려)
- [ ] High 우선순위 기능 다음 업데이트 계획
- [ ] FAQ 업데이트 (자주 묻는 질문 추가)
```

---

### 4단계: 실행 및 후속 조치

**버그 수정:**
1. Critical 버그 → Hotfix (docs/HOTFIX_PROCESS.md)
2. High/Medium 버그 → 다음 정기 업데이트

**기능 개발:**
1. Critical/High 우선순위 → 다음 업데이트 계획
2. Medium/Low 우선순위 → 백로그

**사용자 알림:**
- 수정/개발 완료 시 피드백 제공자에게 알림
- Play 리뷰 답변 업데이트

**예시:**
```
안녕하세요,

말씀하신 [문제/기능]을 수정/추가한 업데이트(v0.X.X)를 출시했습니다.

Play 스토어에서 업데이트 후 확인해 주세요.

피드백 주셔서 감사합니다!

KooDTX 팀
```

## 📊 피드백 분석 도구

### 1. 피드백 트래킹 스프레드시트 (선택사항)

**구조:**

| 날짜 | 채널 | 타입 | 내용 요약 | Priority | Status | 담당자 | 완료일 |
|------|------|------|-----------|---------|--------|--------|--------|
| 2025-01-15 | Play Review | Bug | GPS 정확도 낮음 | High | 진행 중 | John | - |
| 2025-01-16 | Email | Feature | 다크 모드 요청 | Low | 백로그 | - | - |

**활용:**
- 월간/분기별 트렌드 분석
- 가장 많이 요청된 기능 파악

---

### 2. GitHub Issues & Labels

**라벨 체계:**

- **Type**:
  - `bug`: 버그 리포트
  - `enhancement`: 기능 요청/개선
  - `question`: 일반 문의
  - `documentation`: 문서 개선

- **Priority**:
  - `critical`: 즉시 처리 필요
  - `high`: 다음 업데이트
  - `medium`: 여유 시간
  - `low`: 백로그

- **Status**:
  - `triage`: 검토 중
  - `in-progress`: 작업 중
  - `blocked`: 블로킹 이슈 있음
  - `resolved`: 해결 완료

**워크플로우:**
1. 새 피드백 → GitHub Issue 생성
2. Type, Priority 라벨 추가
3. Milestone 할당 (v0.X.X)
4. 개발 시작 → `in-progress` 라벨
5. 완료 → `resolved` 라벨, Issue 닫기

## 💡 피드백 활용 Best Practices

### 1. 빠른 응답

**원칙:**
- 24-48시간 내 초기 응답
- 1-2점 리뷰는 24시간 내 우선 응답

**효과:**
- 사용자 만족도 증가
- 부정적 리뷰를 긍정적으로 전환 가능

---

### 2. 투명한 소통

**원칙:**
- 문제 인정: "불편을 드려 죄송합니다"
- 계획 공유: "다음 업데이트에서 수정 예정입니다"
- 진행 상황 업데이트

**효과:**
- 사용자 신뢰 증가
- 커뮤니티 형성

---

### 3. 정량적 데이터 결합

**원칙:**
- 피드백 + Analytics 데이터 분석
- 예: "GPS 정확도 낮음" 피드백 + "GPS 정확도 평균 30m" 데이터

**효과:**
- 더 정확한 우선순위 결정
- 문제 심각도 파악

---

### 4. 패턴 인식

**원칙:**
- 개별 피드백이 아닌 패턴에 집중
- 3회 이상 반복 → FAQ 추가 또는 기능 개선

**효과:**
- 근본 원인 해결
- FAQ로 지원 부담 감소

---

### 5. 클로즈드 루프

**원칙:**
- 피드백 제공자에게 결과 알림
- "말씀하신 기능을 v0.X.X에 추가했습니다"

**효과:**
- 사용자 참여감 증가
- 충성도 높은 사용자 확보
